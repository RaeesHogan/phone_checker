# Phase 2: Product Requirements Document (PRD)

## 1. Product Overview
**Customer Reservation Lock System** เป็นระบบเว็บแอปพลิเคชันที่ออกแบบมาเพื่อป้องกันการจองซ้ำซ้อนโดยใช้เบอร์โทรศัพท์เป็นกุญแจสำคัญ (Lock Key) ระบบนี้เน้นความเรียบง่าย ปลอดภัย และตรวจสอบได้ โดยอนุญาตให้บุคคลทั่วไปตรวจสอบสถานะได้ แต่จำกัดการเข้าถึงข้อมูลส่วนบุคคลไว้สำหรับผู้ใช้ที่ได้รับอนุญาตเท่านั้น

## 2. Product Goals
- **Eliminate Duplicates:** ป้องกันไม่ให้เกิดการจองซ้ำสำหรับเบอร์โทรศัพท์ที่ยังมีสถานะ ACTIVE
- **Privacy First:** ปกป้องข้อมูลลูกค้าจากการเข้าถึงโดยไม่ได้รับอนุญาต
- **Auditability:** บันทึกทุกกิจกรรมที่เกิดขึ้นในระบบเพื่อความโปร่งใส
- **Ease of Use:** ระบบใช้งานง่ายทั้งบนมือถือและคอมพิวเตอร์

## 3. Success Metrics (KPIs)
- **Zero Duplicate Active Reservations:** ไม่มีการจองซ้ำในระดับฐานข้อมูล
- **System Accuracy:** ระบบอัปเดตสถานะหมดอายุ (14 วัน) ได้อย่างแม่นยำ 100%
- **Performance:** หน้าค้นหาสาธารณะต้องตอบสนองภายใน < 1 วินาที
- **Audit Coverage:** 100% ของการแก้ไขข้อมูลต้องมีการบันทึก Audit Log

## 4. User Roles & Permissions
- **Public (Unauthenticated):** ค้นหาเบอร์โทรศัพท์ได้ (เห็นเฉพาะสถานะ "มีการจองแล้ว" หรือ "ยังไม่มีการจอง")
- **User (Internal Staff):** ล็อกอิน, สร้างการจอง, ดู/แก้ไข/ยกเลิกการจองของตนเอง
- **Admin (Manager):** จัดการผู้ใช้, ดูการจองทั้งหมด, ส่งออกไฟล์ Excel, ดู Audit Logs และ System Logs

## 5. User Journeys
### 5.1 Public User Journey
1. เข้าหน้าแรก -> กรอกเบอร์โทรศัพท์ -> กดค้นหา
2. ระบบแสดงสถานะ "มีการจองแล้ว" (สีแดง) หรือ "ยังไม่มีการจอง" (สีเขียว)
3. หากยังไม่มีการจอง จะมีลิงก์ให้เจ้าหน้าที่ล็อกอินเพื่อเข้าสู่ระบบ

### 5.2 Internal Staff Journey (Creating Reservation)
1. ล็อกอินเข้าระบบ
2. ค้นหาเบอร์โทรศัพท์ที่ต้องการจอง
3. หากยังไม่มีการจอง ระบบจะเปิดฟอร์มการจองพร้อมกรอกเบอร์โทรศัพท์ให้อัตโนมัติ
4. กรอกชื่อลูกค้า, ที่อยู่, รหัสสินค้า -> บันทึกการจอง
5. ระบบบันทึกวันจองและคำนวณวันหมดอายุ (+14 วัน) ให้อัตโนมัติ

### 5.3 Admin Journey (Management & Audit)
1. ล็อกอินด้วยสิทธิ์ Admin
2. เข้าหน้า Dashboard เพื่อดูภาพรวม
3. ตรวจสอบ Audit Logs เมื่อพบความผิดปกติ
4. ส่งออกข้อมูลการจองทั้งหมดเป็น Excel เพื่อนำไปรายงานผล

## 6. Acceptance Criteria
- [ ] ระบบต้องตรวจสอบเบอร์โทรศัพท์ที่ซ้ำกันในสถานะ ACTIVE
- [ ] ข้อมูลลูกค้าต้องไม่ถูกแสดงในหน้าค้นหาสาธารณะ
- [ ] การจองจะหมดอายุอัตโนมัติเมื่อครบ 14 วัน
- [ ] Admin สามารถปิดการใช้งาน (Disable) บัญชีผู้ใช้ได้

## 7. Constraints & Assumptions
- รองรับเบอร์โทรศัพท์ในประเทศไทยเป็นหลัก
- ข้อมูลการจองจะไม่มีการลบทิ้ง (ใช้การเปลี่ยน Status แทน)
- การคำนวณ 14 วัน เริ่มนับจากเวลาที่บันทึกข้อมูลสำเร็จ
